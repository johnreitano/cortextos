# Claude Code Workflows: Use Cases and Patterns

**Date:** 2026-06-29
**Focus:** Practical use cases and reusable patterns for Claude Code Workflows, with emphasis on content pipelines, research workflows, coding workflows, QA/evals, web scraping, long-running agent operations, and Agent Architect examples.

## Executive Summary

Claude Code Dynamic Workflows are best understood as script-level orchestration for work that is too broad, noisy, or repetitive for one conversation to coordinate. The workflow is a JavaScript script that Claude writes, then a background runtime executes. The script controls loops, branching, fan-out, and fan-in, while spawned agents do the actual file reads, edits, shell commands, web fetches, and MCP calls.

The core advantage is that the plan moves out of the chat context and into code. Intermediate results stay in script variables instead of flooding Claude's conversation. That makes workflows strong for codebase-wide audits, multi-source research, large migrations, adversarial verification, content intelligence, recurring QA passes, and long-running operations where dozens or hundreds of agents need to work independently before a synthesis step.

For Agent Architects, the important pattern is not "many agents." It is structured delegation with evidence capture. A good workflow defines shard boundaries, input schemas, output schemas, validation rules, cost limits, retry behavior, and handoff points. Claude can generate the workflow, but the architect is responsible for making the operating model repeatable, observable, and safe.

## Ground Truth Mechanics

Based on the official Claude Code workflows documentation:

- A workflow is a JavaScript orchestration script generated by Claude for the task you describe.
- The runtime executes that script in the background while the main session stays responsive.
- Workflows are meant for codebase audits, large migrations, and cross-checked research.
- Claude can save a useful workflow under `.claude/workflows/` or `~/.claude/workflows/` so it can be run again as a slash command.
- Saved workflows can receive structured `args`, such as a research question, issue list, path list, or config object.
- Workflow scripts coordinate agents but do not directly access the filesystem or shell. Agents perform reads, writes, commands, fetches, and MCP calls.
- Current documented limits are up to 16 concurrent agents and 1,000 total agents per run.
- There is no mid-run user input except permission prompts. If a human checkpoint is required, split the job into multiple workflows.
- Runs are resumable within the same session because the runtime tracks agent results as the run progresses.

Sources: [Claude Code workflows docs](https://code.claude.com/docs/en/workflows), [Claude Code subagents docs](https://code.claude.com/docs/en/sub-agents), [Claude Code hooks docs](https://code.claude.com/docs/en/hooks-guide), [Claude Code skills docs](https://code.claude.com/docs/en/skills).

## Primitive Selection

| Primitive | Best use | Pattern role |
|---|---|---|
| Workflow | Many agents, repeatable orchestration, broad fan-out, scripted aggregation | Top-level coordinator |
| Subagent | Noisy side task, specialized review, isolated context, tool-limited worker | Worker unit |
| Skill | Reusable procedure, domain playbook, checklist, tool recipe | Operating manual |
| Hook | Deterministic behavior at lifecycle points | Guardrail, notification, validation, auto-format |
| Git worktree | Parallel implementation without shared working-tree conflicts | Isolation boundary |
| GitHub Action | Event-triggered coding, review, issue handling, scheduled reports | CI/CD trigger surface |
| MCP tool | External system access, APIs, browser, database, scraping, docs | Tool substrate |
| Agent SDK | Programmatic embedding of Claude Code behavior into an app | Productized orchestration |

Practical rule: use a skill when the procedure is the reusable asset, a subagent when the role is the reusable asset, a hook when deterministic enforcement is needed, and a workflow when the orchestration itself is the reusable asset.

## Reusable Workflow Patterns

### 1. Fan-Out / Fan-In Audit

Split a large surface into shards, run one worker per shard, normalize each result, then synthesize.

Best for:
- Security audits across routes, services, or packages
- Dead-code detection
- Dependency migration impact mapping
- Content opportunity scans across many feeds
- Research claim collection across many sources

Architectural requirements:
- Stable sharding rule, such as file glob, URL list, package name, customer segment, or paper cluster
- Worker prompt with strict output schema
- Aggregator step that deduplicates, ranks, and cites evidence
- Review step that samples negatives, not only positives

### 2. Map-Reduce Synthesis

Run independent workers to extract structured facts, then reduce into a brief, table, ranking, or recommendation.

Best for:
- Literature reviews
- Competitive analysis
- Customer interview synthesis
- Long repository explanation
- Content clustering

Pattern detail:
- Map phase extracts claims, numbers, quotes, risks, and source metadata.
- Reduce phase merges duplicates and separates confirmed claims from single-source claims.
- Optional referee phase challenges the reducer's conclusions.

### 3. Adversarial Verification

Assign one group of agents to produce an answer and another group to disprove, exploit, or falsify it.

Best for:
- High-stakes research briefs
- Security findings
- Coding migrations
- Eval design
- Public-facing content claims

Useful roles:
- Builder
- Skeptic
- Source verifier
- Test runner
- Edge-case finder
- Final arbiter

The key output is not just "passed" or "failed." It is an evidence bundle with reproduction steps, cited sources, test output, and remaining uncertainty.

### 4. Pipeline With Artifacts

Break work into stages where each stage writes an artifact that becomes the next stage's input.

Best for:
- Content pipelines
- Research dossiers
- Scraping and enrichment
- QA reports
- Course or agent training material

Example artifact chain:

```text
source list -> extracted records -> normalized dataset -> scored candidates -> brief -> review notes -> publish package
```

This pattern is safer than a single giant prompt because each stage can be inspected, re-run, and replaced.

### 5. Worktree-Isolated Implementation

Use isolated worktrees for agents that might edit code concurrently.

Best for:
- Parallel refactors
- Multi-module feature implementation
- Competing solution attempts
- Risky migrations
- Automated PR branches

Important caveat: uncommitted main-session changes are not visible to workers spawned from the committed base. Commit, stash, or pass the relevant context explicitly before dispatch.

Local prior research: `docs/research/claude-code-worktrees-2026-04-19.md`.

### 6. Human Checkpoint Boundary

Because workflows do not support arbitrary mid-run user input, design approval points as workflow boundaries.

Pattern:

```text
workflow A: gather and propose -> human approval -> workflow B: execute approved plan -> workflow C: verify and package
```

Use this for:
- External publishing
- Production deploys
- Email or social sending
- Data deletion
- High-cost multi-hour runs

### 7. Deterministic Guardrail Layer

Use hooks for rules that must always run, not just rules Claude should remember.

Examples:
- Auto-format after file edits
- Block edits to protected files
- Validate read-only SQL before database access
- Notify when Claude needs input
- Re-inject context after compaction
- Audit config changes
- Auto-approve low-risk commands from an allowlist

Source: [Claude Code hooks docs](https://code.claude.com/docs/en/hooks-guide).

### 8. Evidence Ledger

Every worker emits source references, file paths, command output summaries, timestamps, and confidence. The reducer preserves this ledger instead of hiding it behind prose.

Best for:
- Research workflows
- Scraping workflows
- QA/eval reports
- Compliance-sensitive automation
- Agent Architect client deliverables

Output schema:

```json
{
  "claim": "string",
  "evidence": ["url or file path"],
  "confidence": "high | medium | low",
  "counterevidence": ["url or file path"],
  "recommended_action": "string"
}
```

## Use Case: Content Pipelines

Claude Code Workflows are a strong fit for content systems because content work is usually a staged pipeline: discovery, extraction, clustering, angle selection, drafting, review, packaging, and distribution.

### Pattern A: Signal-to-Brief Pipeline

Goal: turn many raw signals into a ranked editorial briefing.

Workflow:

1. Ingest signals from RSS, YouTube transcripts, X posts, newsletters, Reddit, Hacker News, GitHub releases, and internal notes.
2. Shard by source type or topic cluster.
3. Extract structured records: source, summary, novelty, audience fit, proof points, controversy, and suggested angle.
4. Cross-check high-scoring items against primary sources.
5. Rank by relevance, timeliness, story strength, and production effort.
6. Output a daily or nightly briefing with recommended topics.

Agent roles:
- Source collector
- Transcript extractor
- Trend clusterer
- Claim verifier
- Editorial strategist
- Final brief writer

Agent Architect relevance:
- This is a repeatable client-facing "content intelligence OS."
- The durable asset is the workflow plus scoring rubric, not a one-off prompt.
- Skills can hold brand voice, audience personas, content formats, and channel constraints.
- Hooks can block publishing without approval and enforce citation requirements.

### Pattern B: Brief-to-Deliverable Pipeline

Goal: turn an approved topic into multiple deliverables.

Workflow:

1. Read approved brief.
2. Generate outline options.
3. Assign agents to script, newsletter, short-form clips, carousel, course page, and community post variants.
4. Run factual review and brand review.
5. Package final outputs with source notes and platform-specific metadata.

Good fit for:
- Agent Architect workshops
- AI tool tutorials
- Course modules
- Founder-led content
- Newsletter operations

Key guardrail: publishing is an external action. The workflow should produce a publish-ready package, then stop for human approval.

### Pattern C: Content Repurposing Factory

Goal: convert long-form source material into channel-specific assets.

Inputs:
- Long video transcript
- Podcast transcript
- Research report
- Webinar recording notes
- Product changelog

Outputs:
- YouTube description
- Newsletter section
- LinkedIn post
- X thread
- Skool/community post
- Short-form clip candidates
- Landing page FAQ

Quality pattern:
- One agent extracts canonical claims.
- Several agents produce channel variants.
- One verifier checks every variant against canonical claims.
- One editor enforces voice and removes unsupported embellishment.

## Use Case: Research Workflows

Research is one of the strongest workflow categories because it benefits from parallel search, independent verification, and source separation.

### Pattern A: Deep Research With Claim Voting

Workflow:

1. Generate search plan from the research question.
2. Dispatch agents by source class: official docs, academic papers, company blogs, GitHub repos, forums, analyst coverage, and news.
3. Each agent emits claims with citations and confidence.
4. Verification agents challenge each claim using independent sources.
5. Reducer marks claims as confirmed, weak, contradicted, or unresolved.
6. Final writer produces the brief with open questions.

Good fit for:
- Technical deep dives
- Competitive analysis
- Market mapping
- Vendor selection
- "What is the real pattern?" investigations

Source support:
- Official workflows docs call out cross-checked research as a primary use case.
- Claude web search docs identify technical documentation search, literature review, citation verification, and response grounding as strong uses for dynamic filtering.

Sources: [Claude Code workflows docs](https://code.claude.com/docs/en/workflows), [Claude web search docs](https://platform.claude.com/docs/en/agents-and-tools/tool-use/web-search-tool).

### Pattern B: Research-to-Knowledge-Base

Goal: make research reusable rather than trapped in a final document.

Workflow:

1. Collect and verify source material.
2. Write a human-readable brief.
3. Extract structured entities, claims, definitions, source links, and decision notes.
4. Ingest into a vector store, graph, or internal knowledge base.
5. Emit "future query handles" so agents know what to ask later.

Agent Architect relevance:
- This is the bridge from research output to institutional memory.
- Clients often need the second half more than the first: "make this retrievable by future agents."

### Pattern C: Multi-Perspective Strategy Review

Goal: draft a plan from several independent angles before committing.

Worker roles:
- Optimist
- Skeptic
- Operator
- Customer advocate
- Finance/cost reviewer
- Security/compliance reviewer
- Implementation planner

Output:
- Best plan
- Rejected alternatives
- Assumptions
- Risks
- First experiment
- Kill criteria

This fits Claude Code Workflows because the point is independent reasoning before synthesis, not a single conversational brainstorm.

## Use Case: Coding Workflows

Coding workflows are the headline use case because Claude Code has native access to files, shell commands, tests, git, MCPs, and GitHub.

### Pattern A: Codebase-Wide Bug Sweep

Workflow:

1. Shard repository by package, service, route group, or ownership area.
2. Each agent searches for a class of bug, such as auth bypass, unsafe parsing, race conditions, missing validation, flaky async handling, or dead code.
3. Findings include file path, line reference, reproduction path, severity, and proposed fix.
4. Verification agents attempt to disprove each finding.
5. Final output ranks issues and proposes implementation order.

Use when:
- A bug pattern appears in one location and likely exists elsewhere.
- Static analysis is too shallow or noisy.
- You need semantic search across architectural patterns.

### Pattern B: Large Migration

Workflow:

1. Inventory affected files.
2. Build migration rules from examples.
3. Run parallel workers by package or file group.
4. Run formatters, type checks, unit tests, and integration tests.
5. Spawn fix-loop agents for failures.
6. Produce a final PR or set of PRs.

Examples:
- Framework version migration
- API client replacement
- Design system component replacement
- Test framework migration
- Language port

The public flagship example is Jarred Sumner's Bun port from Zig to Rust, reported as roughly 750,000 generated Rust lines, 11 days from first commit to merge, and 99.8 percent of the existing test suite passing. Treat this as an exceptional proof point, not a baseline estimate.

Sources: [local prior research](../2026-05-29/claude-workflows-opus-48.md), [DevOps.com coverage](https://devops.com/claude-codes-dynamic-workflows-take-on-the-tasks-that-were-too-big-to-automate/).

### Pattern C: Multi-Agent Implementation Room

Workflow:

1. Planner creates implementation plan.
2. Separate agents implement backend, frontend, tests, docs, and migration scripts in isolated worktrees.
3. Reviewer agents inspect diffs by concern: security, performance, accessibility, maintainability, test coverage.
4. Integrator merges or cherry-picks the best pieces.
5. Verifier runs the full app and test suite.

Recommended primitives:
- Workflows for orchestration
- Worktrees for isolation
- `/run` and `/verify` skills for app-level verification
- Hooks for formatting and protected-file checks
- GitHub Actions for PR review or scheduled automation

Sources: [Claude Code skills docs](https://code.claude.com/docs/en/skills), [Claude Code GitHub Actions docs](https://code.claude.com/docs/en/github-actions).

### Pattern D: GitHub Event Automation

Workflow surfaces:
- `@claude` in issue or PR comments
- Pull request opened or synchronized
- Scheduled daily report
- Security review prompt
- Automated issue triage

The official GitHub Actions docs show examples for PR review, implementing features from issue descriptions, fixing dashboard errors, and scheduled reports summarizing commits and issues.

Source: [Claude Code GitHub Actions docs](https://code.claude.com/docs/en/github-actions).

## Use Case: QA and Evals

QA/eval work is well suited to workflows because good evaluation is multi-perspective: generate cases, run them, inspect failures, minimize reproductions, and compare against acceptance criteria.

### Pattern A: Regression Matrix Runner

Workflow:

1. Generate or load a test matrix.
2. Shard by browser, route, API endpoint, user role, device size, locale, or dataset.
3. Run tests or browser checks.
4. Capture screenshots, logs, console errors, network failures, and diffs.
5. Classify failures as product bug, test bug, flaky environment, or expected behavior.
6. Produce a prioritized QA report.

Agent Architect relevance:
- This is a productizable "QA room" with specialized agents.
- The deliverable is not only bug findings. It includes evidence, reproduction steps, and recommended ownership.

### Pattern B: Prompt and Agent Evals

Workflow:

1. Load tasks, rubrics, expected outputs, and failure examples.
2. Run candidate prompts, skills, agents, or workflow versions against the suite.
3. Use evaluator agents to score factuality, constraint adherence, format adherence, safety, and usefulness.
4. Use adversarial agents to test prompt injection, tool misuse, and edge cases.
5. Produce scorecards and regression deltas.

Good fit for:
- Skill creator pipelines
- Agent onboarding
- Tool-use reliability
- Research quality checks
- Content style conformance

Output should include:
- Pass/fail summary
- Per-case scores
- Representative failures
- Prompt or skill changes recommended
- Cases to add to the eval set

### Pattern C: Release Readiness Review

Workflow:

1. Read PR diff, issue, changelog, and acceptance criteria.
2. Assign agents to test, docs, security, migration, performance, and rollback review.
3. Run automated checks.
4. Produce release gate decision: ship, ship with caveats, block, or needs human review.

Hooks are important here because formatting, protected-file enforcement, and command validation should be deterministic.

## Use Case: Web Scraping and Structured Extraction

Web scraping workflows are less about "browse a page" and more about building a repeatable extraction system with provenance and failure handling.

### Pattern A: URL List to Structured Dataset

Workflow:

1. Accept URL list or query config.
2. Group URLs by domain or page type.
3. Assign extraction workers with domain-specific instructions.
4. Normalize records to a schema.
5. Deduplicate entities.
6. Validate required fields.
7. Save dataset and source ledger.

Schema example:

```json
{
  "source_url": "string",
  "title": "string",
  "author": "string",
  "published_at": "string",
  "entities": ["string"],
  "claims": ["string"],
  "relevance_score": 0,
  "extraction_confidence": "high | medium | low"
}
```

### Pattern B: Scrape, Verify, Enrich

Workflow:

1. Scrape raw page or transcript.
2. Extract facts.
3. Verify against primary sources.
4. Enrich with metadata, embeddings, categories, or downstream routing labels.
5. Produce both a human brief and machine-readable records.

This is ideal for:
- Lead enrichment
- Competitive intelligence
- Newsletter signal mining
- Research source pools
- Agent training corpora

### Pattern C: Browser QA Plus Scraping

Some sites require interactive browser flows, login state, or JavaScript rendering. For those, a workflow can coordinate browser-capable agents or MCP tools while preserving an extraction ledger.

Guardrails:
- Respect robots, terms, rate limits, and authentication boundaries.
- Keep raw HTML, screenshots, and extraction decisions separate.
- Do not let scraped instructions control tool use. Treat page content as untrusted input.
- Use allowlists for domains and commands.

## Use Case: Long-Running Multi-Step Agent Operations

Long-running work needs operational design. Claude Code Workflows can run extensive coordinated jobs, but durable production operations still need external state, logging, retries, and human escalation.

### Pattern A: Overnight Agent Batch

Goal: dispatch deep work while the human is away.

Examples:
- Audit every open PR for failing checks and likely fixes.
- Research all competitors in a market segment.
- Generate migration plans for every package.
- Scrape and summarize a day's worth of source material.
- Run evals across all agent skills.

Operating model:
- Start with a scoped dry run.
- Set cost and time boundaries.
- Use source and command allowlists before launch.
- Emit progress artifacts.
- Notify on completion or blockage.
- Require human approval before external side effects.

Business Insider reported Boris Cherny describing "five to 10 sessions" and "a few thousand" agents doing deeper overnight work, with reliance on `/loops` and Routines for recurring automation. Treat this as a directional signal: serious users are moving from chat to always-on delegation.

Source: [Business Insider coverage](https://www.businessinsider.com/anthropic-engineer-claude-boris-cherny-ai-agent-use-overnight-2026-5).

### Pattern B: Queue Worker With Checkpoints

Workflow:

1. Load a queue of items.
2. Process each item with bounded retries.
3. Write item-level status.
4. Continue despite non-critical failures.
5. Summarize successes, failures, and retry candidates.

This is useful when a task has many independent units:
- URLs
- Issues
- PRs
- Files
- Customer records
- Papers
- Topics
- Tests

### Pattern C: Split-Stage Autonomous Operation

For jobs that need judgment, split into stages:

```text
discover -> propose -> approve -> execute -> verify -> report
```

Do not bury approval in a long workflow. Make approval a hard boundary. That turns autonomy into controlled delegation.

## Design Principles

### Start With the Output Contract

Define the final artifact before defining agents.

Good output contracts include:
- Required sections
- JSON schema if machine-readable
- Citation requirements
- Confidence scale
- Error categories
- Acceptance criteria
- Human approval point

### Shard by Natural Ownership

Bad sharding creates duplicate work and weak synthesis. Good shard boundaries follow the domain:
- Files by package
- URLs by domain
- Papers by field
- Tests by feature
- Customers by segment
- Content by channel
- Logs by service

### Use Independent Review, Not Just More Workers

More agents only help if their work is independent or specialized. For high-stakes outputs, reserve budget for verification agents.

Review patterns:
- Two independent reviewers per critical item
- Red-team agent against final answer
- Source-only verifier that cannot see the draft
- Test-only verifier that runs commands but does not edit
- Arbiter that resolves disagreements with evidence

### Keep External Side Effects Outside the Run

External sends, deploys, deletes, financial actions, and public publishing should happen after a separate approval. Workflows can prepare the action package, but the side effect should be gated.

### Log the Evidence

Every serious workflow should preserve:
- Inputs
- Worker assignments
- Source links
- File paths touched
- Test commands
- Summaries of command output
- Failed shards
- Retry history
- Final confidence

Without this, workflow speed creates review debt.

### Turn Repeats Into Saved Workflows

If the same orchestration will run again, save it as a workflow command. If only the instructions repeat, make a skill. If a role repeats, make a subagent. If a rule must always fire, make a hook.

## Common Failure Modes

| Failure mode | Cause | Mitigation |
|---|---|---|
| Expensive vague run | Scope too broad and no budget | Dry run, max shards, explicit stop rules |
| Duplicate findings | Poor shard boundaries | Stable IDs, dedupe reducer, source ledger |
| Unsupported claims | Weak source discipline | Citation schema and source verifier |
| Mid-run blockage | Needed human judgment inside workflow | Split into staged workflows |
| Edit conflicts | Multiple agents editing same tree | Worktree isolation and path ownership |
| Verification debt | Generation outpaces review | Dedicated verifier agents and sample audits |
| Tool misuse | Page content or prompt asks agent to do unsafe action | Treat external content as untrusted, use hooks and allowlists |
| Context rot | Too much intermediate material in main session | Keep intermediates in workflow artifacts and subagent contexts |
| Non-repeatability | One-off chat orchestration | Save workflow, skill, schemas, and test cases |
